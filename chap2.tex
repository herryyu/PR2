\chapter{Rust之旅}

Rust给类似于本书的图书作者提出了挑战：使该门语言具有鲜明个性的并不是在第一页就展现出的某些具体、惊人的特性，而是
它的所有部分均被设计得可顺畅地协同工作，以实现我们在上一章所列出的目标：安全、高性能的系统编程。该门语言的各子部分与其
它剩余部分相辅相成，有着最合乎情理的存在。

\medskip
因此，与其每次交涉一个语言特性，我们毋宁准备几个小而美的程序导学；每个程序均在上下文中介绍了一些该门语言的更多特性：

\smallskip
\begin{itemize}
    \item 作为热身，我们有一个能通过其命令行参数做简单运算的程序，且带有单元测试。它展示了Rust的核心类型并介绍了特
    型（traits）。 
    \item 接着，我们会构建一个网络服务器。我们将使用一个第三方库来控制HTTP细节，并引入字符串处理，闭包以及错误处理。
    \item 我们的第三个程序将绘制一个美丽的分形图，并将运算分配在多线程之间以提高速度。它包含一个泛型函数示例，其说明了
    如何处理类似于像素缓存的东西，并展现Rust对于并发的支持。
    \item 最后，我们将展示一个强大的命令行工具，它能以正则表达式处理文件。此部分展现了Rust标准库处理文件的工具，以及
    最常用的正则表达式第三方库。
\end{itemize}

\smallskip
Rust通过最小化性能损耗来预防未定义行为的承诺，对该门语言系统各部分设计影响深远，涉及从类似vector和string这样的标准
数据结构，到Rust程序使用第三方库的方式等，对于如何对此管理妥当的细节涵盖于全书上下。而现在，我们想向你说明的是，Rust
是一门相当能干且令人愉悦的语言。

\medskip
当然了，首先你得在你的电脑上安装Rust。

\bigskip
\mksec{rustup与Cargo}
安装Rust的最佳方式是使用\texttt{rustup}。请前往\reditxt{https://rustup.rs}并根据里面的指示操作。

\medskip
或者你可以访问\redtxt{Rust官方网站}以获取Linux、macOS或Windows的预编译包。Rust也被集成在某些操作系统发行版中。我
们更青睐\texttt{rustup}，因为它是管理Rust工具链安装的工具，类似于Ruby的RVM或Node的NVM。比如说，当Rust发布新版本时，
你可以简单地键入\texttt{rustup update}来进行升级。

\medskip
不管怎样，当你安装好之后，有三个新命令应该能在你的命令行中可用：
\begin{minted}{bash}
    $ cargo --version
    cargo 1.49.0 (d00d64df9 2020-12-05)
    $ rustc --version
    rustc 1.49.0 (e1884a8e3 2020-12-29)
    $ rustdoc --version
    rustdoc 1.49.0 (e1884a8e3 2020-12-29)
\end{minted}

\smallskip
这里的\texttt{\$}是命令提示符；在Windows上，它会是\texttt{C:$\backslash$>}或其它类似的东西。我们在这段命令行抄录中
运行了三个已装载的命令，让其各自反馈它是什么版本。我们对这些命令依次解释：

\smallskip
\begin{itemize}
    \item \texttt{cargo}是Rust的编译管理器，包管理器以及通用工具。你可以使用Cargo创建一个新项目，构建并运行你的程序，并
    管理你的代码所依赖的任何外部库。
    \item \texttt{rustc}是Rust编译器。通常我们让Cargo来帮助调用编译器，但是有时直接运行它也很有用。
    \item \texttt{rustdoc}是Rust文档工具。如果你以适当形式在代码注释中编写文档，\texttt{rustdoc}可以根据它们构建出
    有着漂亮格式的HTML页面。就像\texttt{rustc}，我们通常让Cargo为我们运行\texttt{rustdoc}。
\end{itemize}

\smallskip
为方便起见，Cargo能直接创建一个新的Rust包，它包含一些被安排妥当的标准元数据（standard metadata）：
\begin{minted}{bash}

        Created binary (application) 'hello' package
\end{minted}

\smallskip
该命令创建了一个名叫\mytextit{hello}的新包目录，用于构建一个命令行可执行文件。

\end{minted}

可以看到Cargo创建了文件\mytextit{Cargo.toml}来保存该包的元数据。此时这个文件里还没有太多东西：

讨论\texttt{Cargo.toml}文件的细枝末节。

\medskip
对于我们包的使用，Cargo已经通过\texttt{git}版本控制系统设置好了，该系统创建了一个\mytextit{.git}元数据子目录以及\mytextit{.gitignore}文件。
你可以在命令行中向\texttt{cargo new}传递参数\texttt{--vcs none}来让Cargo跳过这一步骤。

\mksec{Rust函数}
Rust的语法被故意设计得不那么具备原创性。如果你熟悉C，C++，Java或者JavaScript，你也许能通过一个Rust程序来找到答案。
下面是一个计算两整形数之间最大公约数的函数，它使用了欧几里得算法（Euclid's algorithm）。你可以将这段代码加到\mytextit{src/main.rs}的末尾：
\begin{minted}{rust}
    fn gcd(mut n: u64, mut m: u64) -> u64 {
        assert!(n != 0 && m != 0);
        while m != 0 {
            if m < n {
                let t = n;
                m = n;
                n = t;
            }
            m = m % n;
        }
        n
    }
\end{minted}

\smallskip
关键字\texttt{fn}（读作``fun''）引入一个函数。这里，我们定义一个叫做\texttt{gcd}的函数，其两个参数\texttt{n}和\texttt{m}的类
型为\texttt{u64}，即无符号64位整数。\texttt{->}符号在返回值类型之前：表示该函数返回一个\texttt{u64}值。四个空格的缩进属于Rust
的标准形式。

\medskip
Rust的机器整型命名反映了它们的大小和符号性：\texttt{i32}是一个32位有符号整数；\texttt{u8}是一个8位无符号整数（被用作``byte''，字
节值），等等。\texttt{isize}与\texttt{usize}类型储存同指针大小的有符号与无符号数，在32位平台上长度为32 bits，在64位平台上长度为
64 bits。Rust同样有两种浮点类型，即\texttt{f32}与\texttt{f64}，它们分别是是IEEE单/双精度浮点类型，就像C和C++里的\texttt{float}
与\texttt{double}。

\medskip
默认情况下，变量只要被初始化，其值便不能被修改，但若将关键字\texttt{mut}（读作``mute'', mutable缩写）置于参数\texttt{n}和\texttt{m}
之前，将允许函数体对它们赋值。实际上，大多数变量都不会被二次赋值；在阅读代码时，含关键字\texttt{mut}的变量确实能起到提示作用。

\medskip
该函数体由一个\texttt{assert!}宏调用起头，以验证两个参数均不为零。\texttt{!}字符将其标记为宏调用，而非函数调用。类似于C和C++里的
\texttt{assert}宏，Rust的\texttt{assert!}会检查其参数是否为true，若非则终止该程序并附带一条包含未通过检查源码的位置信息；此类突
然的终止被称为\mytextit{panic}。无论程序如何被编译，Rust总是会检查断言，而不像C与C++那样能跳过断言。同时存在一个\texttt{debug\_assert!}
宏，当追求程序速度而将其编译时，断言检查会被跳过。

\medskip
该函数的核心是包含\texttt{if}和赋值语句的\texttt{while}循环。不像C与C++，Rust不要求为条件表达式加括号，但需要在它们控制的语句周围
加大括号。

\medskip
一个\texttt{let}语句声明一个局部变量，就像我们函数里的\texttt{t}。只要Rust能推导出\texttt{t}如何被使用，就无须标注它的类型。在该函数中，
适配\texttt{t}的类型仅为与\texttt{n}和\texttt{m}匹配的\texttt{u64}。Rust仅在函数体内进行类型推导：你必须将函数参数和返回值的类型
标注出来，就像上面那样。若想具体说明\texttt{t}的类型，我们可以这么写：
\begin{minted}{rust}
    let t: u64 = m;
\end{minted}

Rust有\texttt{return}语句，但\texttt{gcd}函数并不需要它。如若一个函数体以一个不带分号的表达式作为结尾，那么该表达式即为该函数的返回值。
事实上，任意被大括号包围的代码块均能被用作表达式。比如说，以下为一个打印一条信息并得到\texttt{x.cos()}的值作为其自身值的表达式：
\begin{minted}{rust}
    {
        println!("evaluating cos x");
        x.cos()
    }
\end{minted}

在Rust中，当程序控制流即将离开函数时，典型做法是使用该形式来创建函数的值，然后在函数中间用\texttt{return}关键字显式
地提前返回。

\bigskip
\mksec{编写并运行单元测试}
Rust该门语言内建得有对于测试的简单支持。要测试我们的\texttt{gcd}函数，我们可以将以下代码添加到\mytextit{src/main.rs}
末尾：
\begin{minted}{rust}
    #[test]
    fn test_gcd() {
        assert_eq!(gcd(14, 15), 1);
        assert_eq!(gcd(2 * 3 * 5 * 11 * 17, 
                       3 * 7 * 11 * 13 * 19),
                    3 * 11);
    }
\end{minted}

\smallskip
这里我们定义了名叫\texttt{test\_gcd}的函数，它调用\texttt{gcd}并检验其是否返回正确值。定义顶端的\texttt{\#[test]}
将\texttt{test\_gcd}标记为一个测试函数以使其被正常编译流程忽略掉，但当我们用\texttt{cargo test}命令来运行程序时，它
会被自动包含并调用。我们可以将测试函数放在源代码树的任何位置，置于它们所执行代码的旁边，\texttt{cargo test}会对其自动收
集并全部运行。

\medskip
\texttt{\#[test]}标记是属性（\mytextit{attribute}）的一个示例。属性是一种以额外信息来标注函数和其它声明的开放式系统，就
像C++与C\#中的属性（attributes），或Java里的注解（annotations）。它们被用来控制编译器警告与代码样式检查、有条件地包含需
被编译的代码（类似C与C++里的\#ifdef）、告诉Rust如何与其它语言编写的代码进行交互等等。我们以后将看到更多的属性示例。

\medskip
现在我们已为章首创建的\mytextit{hello}包增添了\texttt{gcd}和\texttt{test\_gcd}定义，并位于该包的某个子树目录下，可以
像下面这样运行测试：
\begin{minted}{bash}
    $ cargo test
        Compiling hello v0.1.0 (/home/jimb/rust/hello)
        Finished test [unoptimized + debuginfo] target(s) in 0.23s
        Running /home/jimb/rust/hello/target/debug/deps/hello-851e09d2c4fdf6f4

    running 1 test
    test test_gcd ... ok

    test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
\end{minted}

\bigskip
\mksec{处理命令行参数}
要让我们的程序接收一系列作为命令行参数的数字并打印它们的最大公约数，可以将\mytextit{src/main.rs}中的\texttt{main}函数替换为以下代码：
\begin{minted}{rust}
    use std::str::FromStr;
    use std::env;

    fn main() {
        let mut numbers = Vec::new();

        for arg in env::args().skip(1) {
            numbers.push(u64::from_str(&arg)
                   .expect("error parsing argument"));
        }

        if numbers.len() == 0 {
            eprintln!("Usage: gcd NUMBER ...");
            std::process::exit(1);
        }

        let mut d = numbers[0];
        for m in &numbers[1..] {
            d = gcd(d, *m);
        }

        println!("The greatest common divisor of {:?} is {}",
                numbers, d);
    }
\end{minted}

\smallskip
这是一段比较长的代码，所以我们一块块地看：
\begin{minted}{rust}
    use std::str::FromStr;
    use std::env;
\end{minted}

\smallskip
第一个\texttt{use}声明将标准库特型（trait）\texttt{FromStr}引入作用域。一个trait是一组方法（methods）的集合，其可被某类型
所实现。任何实现了\texttt{FromStr} trait的类型均有方法\texttt{from\_str}，它将一个字符串解析为该类型本身。\texttt{u64}类型已经
实现了\texttt{from\_str}，我们调用\texttt{u64::from\_str}来解析命令行参数。即便我们从未在该程序其它地方用过\texttt{FromStr}这
个名字，一个trait也必须被引入作用域才能提供方法以便使用。我们将在第十一章中详细讨论traits。

\medskip
第二个\texttt{use}声明引入\texttt{std::env}模块\mytextit{module}，它提供了一些有用的函数和类型，用于与执行环境进行交互，这包含\texttt{args}函数，它
使我们能访问程序的命令行参数。

\medskip
继续来看这段程序的\texttt{main}：
\begin{minted}{rust}
    fn main() {
\end{minted}

\smallskip
我们的\texttt{main}函数不返回值，所以可以简单地删掉参数列表后面的\texttt{->}以及返回类型。
\begin{minted}{rust}
    let mut numbers = Vec::new();
\end{minted}

\smallskip
我们声明了一个可变（mutable）局部变量\texttt{numbers}并将它初始化为一个空\texttt{vector}。\texttt{Vec}是Rust的
可增长向量类型，其与C++的\texttt{std::vector}，Python的列表或JavaScript的数组类似。尽管\texttt{vector}是被设计
具有动态增长与缩小的能力，我们仍必须将此变量标记为\texttt{mut}以便Rust允许我们将数字推入其尾部。

\medskip
\texttt{number}的类型为\texttt{Vec<64>}，即一个装载\texttt{u64}的vector，但和之前一样，我们不用将其写出来，Rust会
为我们推导。部分原因是我们推入该vector的是\texttt{u64}值，而且将它的元素传递给仅接收\texttt{u64}值的\texttt{gcd}。
\begin{minted}{rust}
    for arg in env::args().skip(1) {
\end{minted}

\smallskip
这里我们用一个\texttt{for}循环来处理命令行参数，逐轮地将变量\texttt{arg}设置为各参数并计算循环体。

\medskip
\texttt{std::env}模组的\texttt{args}函数返回一个迭代器\mytextit{iterator}，它能根据需要求出各参数，并在结束时发出指示。
迭代器在Rust中无处不在；标准库包含其它迭代器，它们能取得vector元素、文件的行、通信信道上接收的消息，以及几乎任何适合循环的内容。
Rust迭代器非常高效：编译器通常能将它们转译为功能相同的手写式循环代码。我们将在第十五章展示这是如何运作的，并举一些例子。

\medskip
迭代器除了可配合\texttt{for}循环使用，还包含可直接调用的多种方法。比如说，由\texttt{args}返回且被迭代器取得的首值通常是将
被运行程序的名称。我们想跳过它，则调用迭代器的\texttt{skip}方法生成一个忽略了首值的新迭代器。
\begin{minted}{rust}
    numbers.push(u64::from_str(&arg)
           .expect("error parsing argument"));
\end{minted}

\smallskip
我们调用\texttt{u64::from\_str}来试图将命令行参数\texttt{arg}解析为一个64位无符号整数。\texttt{u64::from\_str}并
不是一个对现有某\texttt{u64}值上调用的普通方法，而是一个与\texttt{u64}类型相关联的函数，其类似于C++和Java的静态函数。
\texttt{from\_str}函数并不直接返回一个\texttt{u64}，而是返回一个\texttt{Result}值以表征解析成功与否。一个\texttt{Result}
值包含两种变体：
\begin{itemize}
    \item 一个写作\texttt{Ok(v)}的值，表示解析成功且\texttt{v}是被取得的值。
    \item 一个写作\texttt{Err(e)}的值，表示解析失败且\texttt{e}是一个解释原因的错误值。
\end{itemize}

执行任何可能失败操作（比如执行输入输出或以其它方式与操作系统交互）的函数，均可返回\texttt{Result}类型，其变体\texttt{Ok}携带成功
的结果------已传输的字节数、已打开的文件等等------其变体\texttt{Err}携带一个错误代码以表征哪里出了问题。不同于大多数现代语言，Rust
没有异常（exceptions）：所有错误均由\texttt{Result}或panic处理，正如第七章中所提到。

\medskip
我们使用\texttt{Result}的\texttt{expect}方法来检查解析是否成功。若结果是一个\texttt{Err(e)}，\texttt{expect}会打印一条
包含对\texttt{e}描述的信息并立即结束程序。然而，若结果是\texttt{Ok(v)}，\texttt{expect}便简单地返回\texttt{v}自身，最终它
能得以被推入我们的数字vector的末尾。
\begin{minted}{rust}
    if numbers.len() == 0 {
        eprintln!("Usage: gcd NUMBER ...");
        std::process::exit(1);
    }
\end{minted}

\smallskip
在空的数字集合中没有最大公约数，所以我们检查vector是否至少有一个元素，否则退出程序并显示错误。我们使用\texttt{eprintln!}宏来将错误
信息写入标准错误输出流（standard error output stream）。
\begin{minted}{rust}
    let mut d = numbers[0];
    for m in &numbers[1..] {
        d = gcd(d, *m);
    }
\end{minted}

\smallskip
该循环将\texttt{d}作为它的返回值，不断更新他以保存我们目前已处理所有数字的最大公约数。依旧，我们必须将\texttt{d}标记为可变，
以便我们在循环中对它赋值。

\medskip
该\texttt{for}循环包含两个令人惊异的点。首先，我们写了\texttt{for m in \&numbers[1..];}操作符\texttt{\&}有什么用？其次，
我们写了\texttt{gcd(d, *m);}\texttt{*m}里的\texttt{*}有什么用？其实这两个细节是相辅相成的。

\medskip
直到现在，我们的代码仅对简单值做了操作，比如适合固定大小内存块的整数。但现在我们即将迭代一个vector，它可能是任意大小------也许非常大。
Rust在处理这类值时很谨慎：它想让程序员来掌控内存消耗，明确每个值的寿命，同时确保及时释放不再需要的内存。

\medskip
所以当我们进行迭代时，我们希望告诉Rust该vector的所有权\mytextit{ownership}应属于\texttt{numbers}；我们仅仅是
在循环中借用\mytextit{borrowing}它的元素。\texttt{\&numbers[1..]}里的\texttt{\&}操作符向vector从头数第二个元素
借得了一个引用\mytextit{reference}。\texttt{for}循环迭代被引用的元素，相继使\texttt{m}借用每个元素。\texttt{*m}
里的\texttt{*}操作符对\texttt{m}进行解引用，并取得它指代的值，且这是下一个我们想传递给\texttt{gcd}的\texttt{u64}。
最终，由于\texttt{numbers}是该vector的权所有者，当\texttt{numbers}于\texttt{main}的末尾脱离作用域时，Rust会自动将其
释放。

\medskip
所有权与引用规则是Rust实现内存管理和安全并发的的关键；我们于第四、五章对它们有详细讨论。想要适应Rust那么你需要熟悉这些规则，但
就对该入门之旅而言，你只需要了解，\texttt{\&x}向\texttt{x}借得一个引用，且\texttt{*r}是\texttt{r}引用所指代的值。

\medskip
继续观察程序：
\begin{minted}{rust}
    println!("The greatest common divisor of {:?} is {}",
             numbers, d);
\end{minted}

\smallskip
在将\texttt{numbers}的元素迭代完毕后，程序便需将结果打印到标准输出流。\texttt{println!}宏接收一个模板字
符串（template string）参数，它将剩余参数的格式化版本替换为出现在模板字符串中的\texttt{\{...\}}形式，并将结果打印至标准输出流。

\medskip
不像C和C++那样要求\texttt{main}函数在程序顺利结束时返回零，或者当出问题时返回一个非零值，Rust假定只要\texttt{main}完全返回，则
程序顺利结束。我们只有在显式地调用类似于\texttt{expect}或者\texttt{std::process::exit}的函数时才会导致程序终止并显示一条错误
状态码。

    # https://doc.rust-lang.org/cargo/reference/manifest.html

\end{minted}

\smallskip

crate的版本。在这里，我们想用\texttt{actix-web} crate的\texttt{1.0.8}版本以及\texttt{serde} crate
的\texttt{1.0}版本。crates.io上很可能有这两个crates的更新版本，但通过确定我们测试过的代码所使用的特定版本
，能确保甚至当包的新版本发布时，我们的代码依然能正确编译。我们会在第八章中对版本管理进行详尽讨论。

\medskip
Crates能指定可选的特性：并非所有用户都需要的部分接口或实现，但它们被携带在crate中是有意义的。\texttt{serde}
 crate提供超简洁的方法处理来自web窗体的数据，不过根据\texttt{serde}的文档，仅当我们选用了该crate的
\texttt{derive}特性时其才可用，所以我们像上面那样在\mytextit{Cargo.toml}中将该特性进行了指定。

\medskip
注意，我们仅需写下想直接使用的crates名称；\texttt{cargo}会负责逐个引入它们所依赖的其它crates。

\medskip
对于第一个版本，我们将保持web服务器尽可能的简洁：它将仅服务于提示用户输入数字来计算的页面。我们会在\texttt{actix-gcd/src/main.rs}
中放置下面的代码：
\begin{minted}{rust}
    use actix_web::{web, App, HttpResponse, HttpServer};    

    fn main() {
        let server = HttpServer::new(||{
            App::new()
                .route("/", web::get().to(get_index()))
        });
        println!("Serving on http://localhost:3000...");
        server
            .bind("127.0.0.1:3000").expect("error binding server to address")
            .run().expect("error running server");
    }

    fn get_index() -> HttpResponse {
        HttpResponse::Ok()
            .content_type("text/html")
            .body(
                r#"
                    <title>GCD Calculator</title>
                    <form action="/gcd" method="post">
                    <input type="text" name="n"/>
                    <input type="text" name="m"/>
                    <button type="submit">Compute GCD</button>
                    </form>
                "#,
            )
    }
\end{minted}

\smallskip
我们以\texttt{use}声明作为开头，这将使\texttt{actix-web} crate的一些定义更容易被我们所获取。当我们写下\texttt{use actix\_web::\{...\}}
时，每个被列在大括号内的名字都能在代码里被直接使用；我们不必每次使用时均写出\texttt{actix-web::HttpResponse}这样的全称，只需简单将其
写作\texttt{HttpResponse}即可。（我们过会儿就会讲到\texttt{serde}）

\medskip
我们的\texttt{main}函数很简单：调用\texttt{HttpServer::new}创建一个能响应单路径``\texttt{/}''请求的服务器；打印一条信息提醒我们如何
连接它；然后将它设置为在本地机器上监听TCP端口3000。

\medskip
我们传递给\texttt{HttpServer::new}的参数是Rust的闭包\mytextit{closure}表达式\texttt{||\{App::new() ... \}}。闭包是可以被调用，
犹如函数一般的一种值。该闭包不接收参数，但它若接收的话，则参数的名称会出现在竖线\texttt{||}之间。\texttt{\{...\}}是该闭包的主体。当
我们启动服务器时，Actix便开启一个线程池来处理即将来临的请求。每个线程调用我们的闭包来获得值\texttt{App}的一个新拷贝，该值告诉它如何对
请求进行路由和处理。

\medskip
该闭包调用\texttt{App::new}来创建一个空的新\texttt{App}并调用其\texttt{route}方法来为路径\texttt{"/"}增加一个单路由。为该路由提供
的处理程序（handler），即\texttt{web::get().to(get\_index)}，通过调用\texttt{get\_index}来处理HTTP \texttt{GET}请求。
\texttt{route}方法返回加持新路由后的\texttt{App}本体。由于该闭包体末尾没有分号，则\texttt{App}是它的返回值，并已准备好被
\texttt{HttpServer}线程所使用。

\medskip
\texttt{get\_index}函数构造一个\texttt{HttpResponse}值来表示对HTTP \texttt{GET}/请求的响应。\texttt{HttpResponse::Ok()}
代表HTTP \texttt{200 OK} 状态，其指代请求成功。我们调用它的\texttt{content\_type}和\texttt{body}方法来增加响应的具体细节；每
次调用均会返回被它们修改后的\texttt{HttpResponse}本体。最后，\texttt{body}的返回值充当\texttt{get\_index}的返回值。

\medskip
因为响应文本包含大量双引号，我们使用Rust的“原始字符串”（raw string）语法来写：字母r，零或者更多哈希标记（即此处的\#字符），一个双引号
，接着是字符串的内容，其由另一个尾随相同数量哈希标记的双引号终止。任何字符都可以不经转义地出现在一个原始字符串中，这包括双引号；实际上，
像“$\backslash$”这样的转义序列并不会被识别。我们随时可以通过在引号周围使用比出现在本文中更多的哈希标记来确保字符串在我们意图的位置结束。

序、将一切链接起来、启动：

\medskip
终于到使用我们于\mytextit{Cargo.toml}中列出的\texttt{serde} crate的时候了：它将提供一个好用的工具来帮我们处理窗体数据。首先，我们需
直接将下面的\texttt{use}语句添加到\mytextit{src/main.rs}的顶端：
\begin{minted}{rust}
    use serde::Deserialize;
\end{minted}

\smallskip
Rust程序员通常将所有的\texttt{use}声明集中在文件顶端，但这不是严格意义上所必须的：Rust允许声明以任何顺序存在，只要它们出现在合适的
嵌套层级即可。

\medskip
下一步，我们来定义一个Rust结构体，它代表我们所期望来自窗体的值：
\begin{minted}{rust}
    #[derive(Deserialize)]
    struct GcdParameters {
        n: u64,
        m: u64,
    }
\end{minted}

\smallskip
这定义了一个名叫\texttt{GcdParameters}的新类型，它有两个字段（fields）\texttt{n}和\texttt{m}，均为\texttt{u64}型---我们的
\texttt{gcd}函数期望的参数类型。

\medskip
该\texttt{struct}定义上方的注解为一个属性（attribute），就像我们之前用来标记测试函数的\texttt{\#[test]}。在一个类型定义上方放
置\texttt{\#[derive(Deserialize)]}属性将告知\texttt{serde} crate在程序被编译时校验该类型，进而自动生成代码来解析一个该类型值，
且该值来自HTML窗体对应\texttt{POST}请求所用格式数据。事实上，该属性足够允许你从几乎任何种类的结构化数据（JSON、YAML、TOML或任意一
种其它文本或二进制格式）中解析出\texttt{GcdParameters}值。\texttt{serde} crate还提供\texttt{Serialize}属性，其生成代码执行
反向操作，即获取Rust值并以结构化形式将它们写出。

\medskip
有该定义后，我们可以着手写处理程序（handler）函数了，很简单：
\begin{minted}{rust}
    fn post_gcd(form: web::Form<GcdParameters>) -> HttpResponse {
        if form.n == 0 || form.m == 0 {
            return HttpResponse::BadRequest()
                .content_type("text/html")
                .body("Computing the GCD with zero is boring.");
        }
        
        let response = 
            format!("The greatest common divisor of the numbers {} and {} \
                    is <b>{}</b>\n",
                    form.n, form.m, gcd(form.n, form.m));

        HttpResponse::Ok()
            .content_type("text/html")
            .body(response)
    }
\end{minted}

\smallskip
对于一个要充当为Actix请求处理程序的函数，其参数必须为Actix知道如何从HTTP请求中提取的类型。我们的
\texttt{post\_gcd}函数接收一个\texttt{form}参数，其类型为\texttt{web::Form::<GcdParameters>}。
Actix知道如何从任意\texttt{web::Form<T>}类型中提取值，当且仅当\texttt{T}可从HTML窗体\texttt{POST}
数据中被反序列化（deserialize）出来。由于我们已将\texttt{\#[derive(Deserialize)]}属性置于\texttt{GcdParameters}类型定义上面，
Actix便能从窗体数据中反序列化它，所以请求处理程序可将一个\texttt{web::Form<GcdParameters>}值作为其参数。这些类型与函数间的关系均会
在编译期被计算出来；若Actix不清楚如何处理你写的处理程序函数的参数，Rust编译器会立即提示你该错误。

\medskip
观察\texttt{post\_gcd}的内部，如果任何一个参数为零，则该函数首先返回一个HTTP \texttt{401 BAD REQUEST}错误，这是因为我们的\texttt{gcd}
函数会因此恐慌（panic）\footnote{译者注：panic在Rust中代表产生严重错误。}。接着，它使用\texttt{format!}宏构建一个对请求的响应。
\texttt{format!}宏同\texttt{println!}宏类似，只是它不是将文本写到标准输出，而是将其作为一个字符串返回。\texttt{post\_gcd}一旦获取
响应中的文本，便会将其包装在一个HTTP \texttt{200 OK}响应中，设置好它的内容类型（content type），并返回给发送端。

\medskip
我们还必须\texttt{post\_gcd}注册为窗体的处理程序。现在我们将\texttt{main}函数替换为以下版本：
\begin{minted}{rust}
    let server = HttpServer::new(||{
        App::new()
            .route("/", web::get().to(get_index))
            .route("/gcd", web::post().to(post_gcd))
    });
    println!("Serving on http://localhost:3000...");
    server
        .bind("127.0.0.1:3000").expect("error binding server to address")
        .run().expect("error running server");
\end{minted}

\smallskip
这里唯一的变化为，我们增添了另一个\texttt{route}调用，通过\texttt{web::post().to(post\_gcd)}来设置路径\texttt{"/gcd"}的处理程序。

\medskip
最后一块拼图便是我们之前写的\texttt{gcd}函数，将它移入\mytextit{actix-gcd/src/main.rs}文件中。该函数就位后，你就可以中断任何可能正在
运行的服务器，重新构建再重启该程序：

\mksec{并发}
Rust很棒的特性之一便是其对并发编程的支持。确保Rust程序摆脱内存错误的相同规则同时也能保证线程间仅能以避免数据竞争的方式共享内存。比如说：

\smallskip
\begin{itemize}
    \item 若使用互斥（mutex）技术来协调多线程对共享数据结构的更改，Rust会确保若非你持有锁（lock），否则将不能访问数据，并在你完成后自动
释放该锁。而在C与C++中，一个锁和它所保护数据之间的关系只能于代码注释中说明。
    \item 若希望在多线程之间共享只读数据，Rust将确保你不会意外地改动数据。在C与C++中，其类型系统有助于解决此问题，但很容易出错。
    \item 若你将数据结构的所有权（ownership）从一个线程转移到另一个线程，Rust会保证你确实出让了对它的所有访问权。在C与C++中，得是你亲自
检查发送所有权的线程上没有东西会再次触碰到数据。如若你没有搞清楚，那么最终效果可能取决于处理器的缓存中恰好有什么，以及你最近对内存做了多少次
写入。
\end{itemize}

\smallskip
在该章节中，我们将引导你完成对第二个多线程程序的编写。

\medskip
你已写好了你的第一个程序：用于实现最大公约数服务器的Actix网络框架是利用线程池来运行请求处理函数。但如果该服务器同时接收多个请求，
它可能会在多个线程中同时运行\texttt{get\_form}和\texttt{post\_gcd}函数。这也许有点让人吃惊，因为我们在编写这些函数时肯定没有考虑到并发性。

\medskip
然而Rust保证，无论你的服务器变得多么复杂，这么做也是安全的：只要你的程序通过编译，那么它就是无数据竞争的。任何Rust函数均为线程安全。

\medskip
该小节的程序将绘制曼德勃罗集（Mandelbrot set\footnote{由Mandelbrot教授于二十世纪七十年代发现}），一种由对复数进行迭代的简单函数生成的分形
图。Mandelbrot集的绘制经常被叫做高度并行\mytextit{embarrassingly parallel}算法，这是因为其所要求的多线程通信模式极其简单；
我们将在\redtxt{第十九章}介绍更多复杂的模式，不过该小节的任务将提前展现一些要点。

\end{minted}

\subsection{什么是Mandelbrot集}
在你阅读代码时，对于它的功能性有一个具体的概念是很有帮助的，所以让我们对一些纯数学进行一次短暂的游历。我们将从一个简单案例开始，然后再增加复杂
的细节，直到我们触及Mandelbrot集的核心计算。

\medskip
下面是一个无限循环，由Rust专有语法编写，即一个\texttt{loop}语句：
\begin{minted}{rust}
    fn square_loop(mut x: f64) {
        loop {
            x = x * x;
        }
    }
\end{minted}

\smallskip
在实际环境中，Rust可将\texttt{x}视作不会用来做任何事，因此可能懒得计算其值。不过现在假设该代码按照编写的方式运行，\texttt{x}的值会发生什
么变化呢？对任何小于1的数字进行平方会使其变得更小，那么它会接近于零；1的平方得到1；对大于1的数字进行平方会使其变大，那么它会接近于正无穷；对
负数进行平方使其成为正数，其之后的计算表现便会遵循以上的规则（\redtxt{图2-3}）。

\smallskip
\begin{figure}[h]
    \centering
    \includegraphics[scale=0.29]{2-3}
    \caption{重复对一个数执行平方运算的效果}
\end{figure}

\smallskip
所以，根据你传递给\texttt{square\_loop}的值，\texttt{x}在循环中的情况将分为：保持为零或1、接近0，以及接近正无穷。

\medskip
现在来考虑一个稍有不同的循环：
\begin{minted}{rust}
    fn square_loop(c: f64) {
        let mut x = 0;
        loop {
            x = x * x + c;
        }
    }
\end{minted}

\smallskip
这一次，\texttt{x}初始为零，我们在每次迭代中对它执行平方再加\texttt{c}来进行调整。这会使我们更难看出\texttt{x}的变化规律，不过一些实验
表明，若\texttt{c}大于0.25或小于-2.0，则\texttt{x}最终会变得无穷大；否则它会落在零的领域某处。

\medskip
下一个问题：我们不使用\texttt{f64}值，而是考虑使用复数的相同循环。crates.io上的\texttt{num} crate中有可供我们使用的复数类型，所以我们
必须在程序的\mytextit{Cargo.toml}文件中的\texttt{dependencies}区域中为\texttt{num}加上一行。以下是目前为止的整个文件内容（我们会在
之后增加更多细节）：
\begin{minted}{toml}
    [package]
    name = "mandelbrot"
    version = "0.1.0"
    authors = ["You <you@example.com>"]
    edition = "2018"
    
    # See more keys and their definitions at 
    # https://doc.rust-lang.org/cargo/reference/manifest.html
    
    [dependencies]
    num = "0.4"
\end{minted}

\smallskip
现在我们可以来写循环的倒数第二个版本：
\begin{minted}{rust}
    use num::Complex;

    fn complex_square_add_loop(c: Complex<f64>) {
        let mut z = Complex { re: 0.0, im: 0.0 };
        loop {
            z = z * z + c;
        }
    }
\end{minted}

\smallskip
将\texttt{z}用作复数是一个习惯做法，所以我们对循环变量进行了重命名。表达式\texttt{Complex \{ re: 0.0, im: 0.0 \}}是我们用
\texttt{num} crate的\texttt{Complex}类型来写复数零的方式。\texttt{Complex}是一个Rust结构化类型（或者说结构体，\mytextit{struct}），
其定义类似于下面的伪代码：
\begin{minted}{rust}
    struct Complex<T> {
        /// Real portion of the complex numbers
        re: T,

        /// Imaginary portion of the complex numbers
        im: T,
    }
\end{minted}

\smallskip
上面的代码定义了一个名叫\texttt{Complex}的结构体，它有两个字段，\texttt{re}和\texttt{im}。\texttt{Complex}是一个泛型
\mytextit{generic}结构：你可以将类型名之后的\texttt{<T>}读作“对于任何类型T”。比如说，\texttt{Complex<f64>}是一个字段\texttt{re}与
\texttt{im}均为\texttt{f64}值的复数类型，\texttt{Complex<f32>}则使用32位浮点型，等等。有了该定义，像
\texttt{Complex \{ re: 0.24, im: 0.3 \}}这样的表达式会产生一个\texttt{Complex}值，其\texttt{re}字段被初始化为0.24，且其\texttt{im}
字段被初始化为0.3。

\medskip
\texttt{num} crate提供了*，+以及其它算术运算符来处理\texttt{Complex}值，因此，该函数的其余部分就像先前版本一样运作，只是它操作的是复平面
上的点，而不仅是实数轴上的点。我们会在\redtxt{第十二章}介绍如何使Rust操作符与你自己的类型配合工作。

\medskip
最后，我们到达了纯数学游历的终点站。Mandelbrot集被定义为复数\texttt{c}的集合，且对\texttt{c}而言，其不会导致\texttt{z}的计算一路飞向无穷。我
们最初的简易平方循环是能被预测的：任何大于1或小于-1的数最终飞向无穷。将\texttt{+ c}扔进每次迭代使得计算行为变得更难以预测：正如我们之前所说，
\texttt{c}的值若大于0.25或小于-2会致使\texttt{z}飞向无穷。然而将这个游戏展开到复数领域后。将产生怪异又美丽的模式，这正是我们想要绘制的。

\medskip
由于复数\texttt{c}同时有实部\texttt{c.re}和虚部\texttt{c.im}，我们会将它们视为笛卡尔平面上一个点的\texttt{x}和\texttt{y}坐标，并且当
\texttt{c}落在Mandelbrot集里时将点涂成黑色，否则为更轻的颜色。所以对于图像里的每个像素，我们必须为每个复平面上的对应点运行之前的循环，看看它是否
会逃逸到无穷远或者永远在原点附近盘旋，然后为它涂上相应的颜色。

\medskip
无限循环需要运行一段时间，不过对于没耐心的人，有两个窍门。第一，如果我们放弃永远运行循环，而仅尝试有限的迭代次数，事实证明，我们依旧能得到对该集
的良好近似。具体需要多少次迭代则取决于我们想让图形边界的绘制有多精确。第二，已经证明的是，若\texttt{z}在某次离开过以原点为圆心半径为2的圆，它
最终肯定会飞离原点无穷远。所以以下是我们循环的最终版本，并且是我们的程序核心：
\begin{minted}{rust}
    use num::Complex;

    /// 尝试确定'c'是否在Mandelbrot集中，用至多'limit'次迭代来决定。
    /// 
    /// 若'c'不属于该集，返回'Some(i)'，此处'i'为当'c'离开圆心为原点半径为2的圆时的迭代次数。
    /// 若'c'似乎为该集成员（更确切地说，当达到迭代限制次数时依然不能证明'c'不是该集成员），则
    /// 返回'None'。
    fn escape_time(c: Complex<f64>, limit: usize) -> Option<usize> {
        let mut z = Complex { re: 0.0, im: 0.0 };
        for i in 0..limit {
            if z.norm_sqr() > 4.0 {
                return Some(i);
            }
            z = z * z + c;
        }
        None
    }
\end{minted}

\smallskip
该函数接收一个复数\texttt{c}，我们欲测试其是否为Mandelbrot集成员；以及在结束测试并声明\texttt{c}可能是一个成员之前要尝试的迭代次数限制。

\medskip
该函数的返回值是一个\texttt{Option<usize>}。Rust的标准库按如下所示来定义\texttt{Option}类型：
\begin{minted}{rust}
    enum Option<T> {
        None,
        Some(T),
    }
\end{minted}

\smallskip
\texttt{Option}是一个枚举类型\mytextit{enumerated type}，通常称为\mytextit{enum}，因为它的定义列举了该类型值几种可能的变体：对于任何
类型\texttt{T}，一个\texttt{Option<T>}类型的值要么是\texttt{Some(v)}，其中\texttt{v}是一个\texttt{T}类型的值；要么是\texttt{None}
，表示没有\texttt{T}值可用。\texttt{Option}如同我们之前讨论的\texttt{Complex}，是一个泛型类型：你可以用\texttt{Option<T>}代表你喜
欢的任何类型\texttt{T}的一个可选值。

\medskip
在我们的例子中，\texttt{escape\_time}返回一个\texttt{Option<usize>}来表征\texttt{c}是否在Mandelbrot集中---若不在，则我们要迭代多久
才能得出结果。如果\texttt{c}不在该集里，\texttt{escape\_time}返回\texttt{Some(i)}，其中\texttt{i}为当\texttt{z}离开半径为2的圆
时所迭代的次数。否则，\texttt{c}显然在该集中，并且\texttt{escape\_time}此时返回\texttt{None}。
\begin{minted}{rust}
    for i in 0..limit {
\end{minted}

\smallskip
之前的例子展示了\texttt{for}循环对命令行参数和vector元素的迭代；这里的\texttt{for}循环只是简单地对从\texttt{0}到（但是不包括）
\texttt{limit}范围内的整数进行迭代。

\medskip
\texttt{z.norm\_sqr()}方法返回\texttt{z}到原点距离的平方。要判断\texttt{z}是否已经离开了半径为2的圆，我们不用计算平方根，只需将距离的
平方与4.0进行比较，这样更快。

\medskip
你或许有注意到我们使用 \texttt{///} 来标记函数上方的注释行；\texttt{Complex}结构体成员上方的注释同样以 \texttt{///} 起头。它们被称为
文档注释\mytextit{documentation comments}；实用程序\texttt{rustdoc}清楚如何对它们以及它们所描述的代码进行解析，并生成在线文档。Rust
标准库的文档就是以这种形式来写的。我们在\redtxt{第八章}中对文档注释有详细介绍。

\subsection{解析命令行参数对}
该程序接收多个命令行参数来控制我们写入图像的分辨率，以及该图像要显示的Mandelbrot集部分。由于这些命令行参数均遵循一个共同的形式，我们用下面的函数
来解析它们：
\begin{minted}{rust}
    /// 将字符串's'解析为一个坐标对，像是'"400x600"'或'"1.0,0.5"'
    ///
    /// 具体而言，'s'的形式应为<left><sep><right>，<sep>为参数'separator'（分隔符）给定的字符，
    /// <left>和<right>均为可由'T::from_str'解析的字符串。'separator'必须是一个ASCII字符。
    /// 
    /// 若's'的形式正确，返回'Some<(x, y)>'，若不能正确解析，则返回'None'。
    fn parse_pair<T: FromStr>(s: &str, separator: char) -> Option<(T, T)> {
        match s.find(separator) {
            None => None,
            Some(index) => {
                match (T::from_str(&s[..index]), T::from_str(&s[index+1..])) {
                    (Ok(l), Ok(r)) => Some((l, r)),
                    _ => None
                }
            }
        }
    }

    #[test]
    fn test_parse_pair() {
        assert_eq!(parse_pair::<i32>("",        ','), None);
        assert_eq!(parse_pair::<i32>("10",      ','), None);
        assert_eq!(parse_pair::<i32>(",10",     ','), None);
        assert_eq!(parse_pair::<i32>("10,20",   ','), Some((10, 20)));
        assert_eq!(parse_pair::<i32>("10,20xy", ','), None);
        assert_eq!(parse_pair::<f64>("0.5x",    'x'), None);
        assert_eq!(parse_pair::<f64>("0.5x1.5", 'x'), Some((0.5, 1.5)));
    }
\end{minted}

\smallskip
\texttt{parse\_pair}的定义为一个泛型\mytextit{generic}函数：
\begin{minted}{rust}
    fn parse_pair<T: FromStr>(s: &str, separator: char) -> Option<(T, T)> {
\end{minted}

\smallskip
你可以把子句\texttt{<T: FromStr>}大声读作：“对于任何实现了\texttt{FromStr} trait的类型\texttt{T}...”。这能有效地让我们一次性定义
整个函数族：\texttt{parse\_pair::<i32>}是一个解析\texttt{i32}值对的函数，\texttt{parse\_pair::<f64>}解析浮点值对，等等。这非常
像C++里的函数模板。Rust程序员会将\texttt{T}称为\texttt{parse\_pair}的类型参数\mytextit{type parameters}。当使用泛型函数时，Rust
通常能够为你推导类型参数，并且你不需像测试代码那样将类型写出来。

\medskip
我们的返回类型为\texttt{Option<(T, T)>}：不是\texttt{None}就是一个\texttt{Some((v1, v2))}值，其中\texttt{(v1, v2)}是一个包含
两个值（类型均为\texttt{T}）的tuple（元组）。\texttt{parse\_pair}函数并未使用显式的return语句，它的返回值为其函数体内的最后一个（也
是唯一的）表达式：
\begin{minted}{rust}
    match s.find(separator) {
        None => None,
        Some(index) => {
            ...
        }
    }
\end{minted}

\smallskip
\texttt{String}类型的\texttt{find}方法在字符串内搜索与\texttt{separator}匹配的字符。若\texttt{find}返回\texttt{None}，意味着分隔符
并未出现在字符串里，此时整个\texttt{match}表达式的计算结果为\texttt{None}，表示解析失败。否则，我们取得\texttt{index}作为分隔符在字符串
中的位置。
\begin{minted}{rust}
    match (T::from_str(&s[..index]), T::from_str(&s[index+1..])) {
        (Ok(l), Ok(r)) => Some((l, r)),
        _ => None
    }
\end{minted}

\smallskip
这段代码开始展现\texttt{match}表达式的威力了。match的参数为一个元组表达式：
\begin{minted}{rust}
    (T::from_str(&s[..index]), T::from_str(&s[index+1..]))
\end{minted}

\smallskip
表达式\texttt{\&s[..index]}和\texttt{\&s[index+1..]}均为字符串的切片，分别在分隔符前后。类型参数\texttt{T}的关联函数
\texttt{from\_str}分别接收它们，并尝试将它们解析为\texttt{T}类型值，最后生成一个包含结果的元组。以下为我们欲匹配的模式（pattern）：
\begin{minted}{rust}
    (Ok(l), Ok(r)) => Some((l, r)),
\end{minted}

\smallskip
该模式仅当元组内的元素均为\texttt{Result}类型的\texttt{Ok}变体时才算匹配成功，并表示它们均解析成功。如果匹配，\texttt{Some((l, r))}便
为该match表达式的值，因此是该函数的返回值。
\begin{minted}{rust}
    _ => None
\end{minted}

\smallskip
下划线通配符\texttt{\_}匹配任何其它东西并忽略其值。如果我们到达此处，则\texttt{parse\_pair}失败，因此我们将其算为\texttt{None}，并作为
函数返回值。

\medskip
现在我们有了\texttt{parse\_pair}，要写一个解析一对浮点坐标并将它们作为\texttt{Complex<f64>}值返回的函数十分容易：
\begin{minted}{rust}
    /// 将一对由英文逗号隔开的浮点数解析为一个复数
    fn parse_complex(s: &str) -> Option<Complex<f64>> {
        match parse_pair(s, ',') {
            Some((re, im)) => Some(Complex { re, im }),
            None => None
        }
    }

    #[test]
    fn test_parse_complex() {
        assert_eq!(parse_complex("1.25,-0.0625"),
                Some(Complex { re: 1.25, im: -0.0625 }));
        assert_eq!(parse_complex(",-0.0625"), None);
    }
\end{minted}

\smallskip
\texttt{parse\_complex}函数调用\texttt{parse\_pair}，当坐标解析成功时构造一个\texttt{Complex}值，否则失败值\texttt{None}传给其
调用者。

\medskip
如果你读得很仔细，你或许注意到我们用了简写形式来构造\texttt{Complex}值。常见做法是以相同名称的变量来对结构体的字段进行初始化，所以与其强迫
你写为\texttt{Complex \{ re: re, im: im \}}，Rust允许你简单地将其写作\texttt{Complex \{ re, im \}}。这种机制是基于JavaScript和
Haskell中的相似符号进行建模的。

\subsection{将像素映射到复数}
我们的程序需要在两个关联坐标空间中运作：输出图像中的每个像素对应着复平面上的一个点。这两个空间之间的关系取决于我们要绘制Mandelbrot集的哪个部分，
以及所要求的图像分辨率，这由命令函参数来确定。下面的函数负责将图像空间\mytextit{image space}变换到
复数空间\mytextit{complex number space}：
\begin{minted}{rust}
    /// 给定输出图像中一个像素的行与列，返回对应的复平面点。
    /// 
    /// 'bounds'为一个给定像素图的宽与高的值对。
    /// 'pixels'为一个(column, row)值对，表示图像中的一个单独像素。
    /// 参数'upper_left'和'lower_right'分别为复平面上的两个点，指明我们的图像覆盖的范围
    fn pixel_to_point(bounds: (usize, usize),
                    pixel: (usize, usize),
                    upper_left: Complex<f64>,
                    lower_right: Complex<f64>) 
        -> Complex<f64>
    {
        let (width, height) = (lower_right.re - upper_left.re,
                                    upper_left.im - lower_right.im);
        Complex {
            re: upper_left.re + pixel.0 as f64 * width / bounds.0 as f64,
            im: upper_left.im - pixel.1 as f64 * height / bounds.1 as f64
            // 这里为什么是减法呢？因为往下时pixel.1会增加，然而往上时虚部会增加。
        }
    }

    #[test]
    fn test_pixel_to_point() {
        assert_eq!(pixel_to_point((100, 200), (25, 175), 
                                Complex { re: -1.0, im:  1.0 },
                                Complex { re:  1.0, im: -1.0 }),
                Complex { re: -0.5, im: -0.75 });
    }
\end{minted}

\smallskip
\redtxt{图2-4}描绘了\texttt{pixel\_to\_point}执行的计算。

\medskip
位于\texttt{pixel\_to\_point}中的代码纯粹是计算，我们就不过多解释了。不过，有几件事需要指出。此形式的表达式引用了元组元素：
\begin{minted}{rust}
    pixel.0
\end{minted}

\smallskip
它引用了元组\texttt{pixel}的第一个元素。
\begin{minted}{rust}
    pixel.0 as f64
\end{minted}

\smallskip
这便是Rust的类型转换语法：它将\texttt{pixel.0}转为\texttt{f64}值。不像C与C++，Rust一般拒绝在数值类型之间执行隐式转换；你必须写明你
需要的转换。这可能很乏味，但是明确在何时发生了何种转换是相当有用的。隐式整数转换看起来没什么罪过，但纵观历史，它们是现实世界中C与C++代码中
频繁出现的bugs和安全漏洞的源头。

\subsection{集合的绘制}
要绘制Mandelbrot集，对于图像中的每个像素，我们只需为其复平面上的对应点应用\texttt{escape\_time}，然后根据结果
对该像素着色：
\begin{minted}{rust}
    /// 将Mandelbrot集矩形渲染到像素缓冲区。
    /// 
    /// 参数'bounds'给出缓冲区'pixels'的宽与高，该缓冲区的每个字节容纳一个灰度像素。
    /// 参数'upper_left'和'lower_right'指定两个复平面点，分别对应像素缓冲区的左上角和右下角。
    fn render(pixels: &mut [u8],
            bounds: (usize, usize),
            upper_left: Complex<f64>,
            lower_right: Complex<f64>) 
    {
        assert!(pixels.len() == bounds.0 * bounds.1);

        for row in 0..bounds.1 {
            for column in 0..bounds.0 {
                let point = pixel_to_point(bounds, (column, row),
                                                    upper_left, lower_right);
                pixels[row * bound.0 + column] = 
                    match escape_time(point, 255) {
                        None => 0,
                        Some(count) => 255 - count as u8
                    };
            }
        }
    }
\end{minted}

\smallskip
目前为止一切都看起来很熟悉。
\begin{minted}{rust}
    pixels[row * bound.0 + column] = 
        match escape_time(point, 255) {
            None => 0,
            Some(count) => 255 - count as u8
        };
\end{minted}

\smallskip
若\texttt{escape\_time}表明\texttt{point}属于set，\texttt{render}便将相应的像素着色为black\texttt{(0)}。否则，
\texttt{render}会依据迭代次数指定更深的颜色。

\subsection{写入图像文件}
\texttt{image} crate提供对广泛图像格式进行读写的函数，以及一些基础图像操作函数。特别是，它包含一个PNG图像文件格式编码器，我们的
程序会用其保存计算的最终结果。要使用\texttt{image}，请将下面一行添加到\mytextit{Cargo.toml}的\texttt{[dependencies]}区域：
\begin{minted}{toml}
    image = "0.13.0"
\end{minted}

\smallskip
准备好后，我们可以写：
\begin{minted}{rust}
    /// 将尺寸由'bounds'给定的缓冲区'pixels'写入名叫'filename'的文件。
    fn write_image(filename: &str, pixels: &[u8], bounds: (usize, usize))
        -> Result<(), std::io::Error> 
    {
        let output = File::create(filename)?;
        
        let encoder = PNGEncoder::new(output);
        encoder.encode(&pixels,
                       bounds.0 as u32, bounds.1 as u32,
                       ColorType::Gray(8))?;
        Ok(())
    }
\end{minted}

\smallskip
该函数进行的操作十分简单：打开一个文件并尝试将图像写入其中。我们将来自\texttt{pixels}的实际像素数据传递给编码器，然后是其来自
\texttt{bounds}的宽与高，最后一个参数说明如何解释\texttt{pixels}中的字节：值\texttt{ColorType::Gray(8)}表示每个字节均为
一个8-bit的灰度值。

\medskip
该函数有趣之处在于它是如何应付出错的。如果遇到一个错误，我们需将其报告并返回给调用者。正如我们之前提到，Rust中可能会失败的函数应
返回一个\texttt{Result}值，它要么是\texttt{Ok(s)}，这里\texttt{s}是成功值，要么是\texttt{Err(e)}，这里\texttt{e}是一
个错误代码。那么\texttt{write\_image}的成功和失败类型分别是什么呢？

\medskip
如果一切顺利，我们的\texttt{write\_image}函数不会返回有用值，此时它将所有有趣的东西写入文件里。所以它的成功类型为单元
\mytextit{unit}类型\texttt{()}，之所以这么取名是因为它仅有一个值，也被写作\texttt{()}。单元类型与C和C++里的
\texttt{void}类似。

\medskip
如果发生了错误，则原因要么是\texttt{File::create}无法创建文件，要么是\texttt{encoder.encode}无法将图像写入创建的
文件；此时I/O操作会返回一个错误代码。\texttt{File::create}的返回类型是
\texttt{Result<std::fs::File, std::io::Error>}，同时\texttt{encoder.encode}的返回类型为
\texttt{Result<(), std::io::Error>}，它们均共享错误类型\texttt{<std::io::Error>}，则我们的
\texttt{write\_image}函数也返回该错误类型就说得通了。无论出现哪种情况，程序失败应立即返回，并传递描述出现何种问题的
\texttt{std::io::Error}值。

\medskip
所以，为了正确地处理\texttt{File::create}的结果，我们需要对它的返回值进行\texttt{match}，就像这样：
\begin{minted}{rust}
    let output = match File::create(filename) {
        Ok(f) => f,
        Err(e) => {
            return Err(e);
        }
    };
\end{minted}

\smallskip
成功时，将\texttt{output}绑定为携带\texttt{Ok}值的\texttt{File}。失败时，将错误返回给调用者。

\medskip
这种\texttt{match}语句在Rust中是如此常见的一种模式，以致该语言为其提供了\texttt{?}操作符作为简写。所以，与其
每次在尝试可能会失败的操作时显式地套用该逻辑，你可以使用以下等同且更清晰的语句：
\begin{minted}{rust}
    let output = File::create(filename)?;
\end{minted}

\smallskip
若\texttt{File::create}失败，操作符\texttt{?}会让\texttt{writes\_image}返回错误。否则，\texttt{output}持有
成功打开的\texttt{File}。

\smallskip
\mknote{初学者的常见错误是试图在\texttt{main}函数中使用\texttt{?}。然而，由于\texttt{main}本身不返回任何值，
这种写法无效；相反，你需使用\texttt{match}语句，或者像\texttt{unwrap}和\texttt{expect}之一的简写方法。还有一种
选项，只需将\texttt{main}改为返回一个\texttt{Result}，我们会在之后对其讨论。}

\subsection{一个并发Mandelbrot程序}
所有拼图均已就位，我们可以向你展示\texttt{main}函数，我们将在此加入并发。首先，下面是一个非并发的简单版本：
\begin{minted}{rust}
    use std::env;

    fn main() {
        let args: Vec<String> = env::args().collect();

        if args.len() != 5 {
            eprintln!("Usage: {} FILE PIXELS UPPERLEFT LOWERRIGHT",
                    args[0]);
            eprintln!("Example: {} mandel.png 1000x750 -1.20,0.35 -1,0.20",
                    args[0]);
            std::process::exit(1);
        }

        let bounds = parse_pair(&args[2], 'x')
            .expect("error parsing image dimensions");
        let upper_left = parse_complex(&args[3])
            .expect("error parsing upper left corner point");
        let lower_right = parse_complex(&args[4])
            .expect("error parsing lower right corner point");

        let mut pixels = vec![0; bounds.0 * bounds.1];

        render(&mut pixels, bounds, upper_left, lower_right);

        write_image(&args[1], &pixels, bounds)
            .expect("error writing PNG file");
    }
\end{minted}

\smallskip
将命令行参数收集进一个元素为\texttt{String}的数组中后，我们对它们一一解析然后开始计算。
所有拼图均已就位，我们可以向你展示\texttt{main}函数，并将在此加入并发。首先，下面是一个非并发的简单版本：
\begin{minted}{rust}

\end{minted}

\smallskip
宏调用\texttt{vec![v; n]}创建一个长度为\texttt{n}的vector，其内部元素被初始化为\texttt{v}。则上面的代码创建
一个元素为零，长度为\texttt{bounds.0 * bounds.1}的vector，其中\texttt{bounds}为由命令行解析而来的图像分辨
率。我们会将此vector用作一个矩形数组，其元素为单字节灰度像素值，如\redtxt{图2-5}所示。

下一个关注点是：

\end{minted}

\smallskip
它调用了\texttt{render}函数来对图像进行实际计算。表达式\texttt{\&mut pixels}借用了像素缓冲区的一个可变引用，允许

    write_image(&args[1], &pixels, bounds)

\end{minted}

最后，我们将像素缓冲区以PNG文件的形式写入磁盘。此时，我们传递一个指向缓冲区的共享引用（非可变），因为
\texttt{write\_image}理应不需要修改缓冲区的内容。

在先前的命令行抄录中，我们使用了Unix的\texttt{time}程序来分析我们的程序运行时间：它花了约整整五秒钟对图像的每个像素

能提供的所有计算资源，我们应该能极大地加速图像计算。

\medskip
为此，我们将把图像分割为数个区域，为它们各分配一个处理器，然后让各处理器对分配给它的像素进行着色。为方便起见，我们将图像
划分为一系列水平条带，如\redtxt{图2-7}所示。当所有处理器完成任务时，我们便可将像素写入磁盘。

\smallskip
\begin{figure}[h]
    \centering
    \includegraphics[scale=0.29]{2-7}
    \caption{将像素缓冲区分割为数个条带以方便并行渲染}
\end{figure}

\smallskip
第三方\texttt{crossbeam} crate提供大量有用的并发功能，包括作用域线程\mytextit{scoped thread}功能，这正是我们所
需要的。要使用它，我们必须将以下一行添加到\mytextit{Cargo.toml}文件中：
\begin{minted}{toml}
    crossbeam = "0.8"
\end{minted}

\smallskip
下一步我们需去除调用\texttt{render}的单独一行并将其替换为下面的代码：
\begin{minted}{rust}
    let threads = 8;
    let rows_per_band = bounds.1 / threads + 1;

    {
        let bands: Vec<&mut [u8]> = 
            pixels.chunks_mut(rows_per_band * bounds.0).collect();
        crossbeam::scope(|spawner| {
            for (i, band) in bands.into_iter().enumerate() {
                let top = rows_per_band * i;
                let height = band.len() / bounds.0;
                let band_bounds = (bounds.0, height);
                let band_upper_left = 
                    pixel_to_point(bounds, (0, top), upper_left, lower_right);
                let band_lower_right = 
                    pixel_to_point(bounds, (bounds.0, top + height),
                                   upper_left, lower_right);
                spawner.spawn(move |_| {
                    render(band, band_bounds, band_upper_left, band_lower_right);
                });
            }
        }).unwrap();
    }
\end{minted}

\smallskip
现在来一步步对其进行分解：
\begin{minted}{rust}
    let threads = 8;
    let rows_per_band = bounds.1 / threads + 1;
\end{minted}

\smallskip
这里，我们决定使用八个线程\footnote{\texttt{num\_cpus} crate提供一个函数，其返回当前系统可用的CPU核心数}。接着我
们计算各条带应该有多少行像素。我们对行数进行向上取整以确保所有条带能覆盖整个图像（即使图像高度不是\texttt{threads}的倍
数）。
\begin{minted}{rust}
    let bands: Vec<&mut [u8]> = 
        pixels.chunks_mut(rows_per_band * bounds.0).collect();
\end{minted}

\smallskip
此处我们将像素缓冲区分割为一系列条带。缓冲区的\texttt{chunks\_mut}方法返回一个迭代器，它取得该缓冲区的数个可变且不重
叠的切片，它们各自包围了\texttt{rows\_per\_band * bounds.0}个像素---换句话说，\texttt{rows\_per\_band}表示
单个条带的像素行数。方法\texttt{chunks\_mut}取得的最后一个切片可能包含更少的行，但是每一行仍包含相同数量的像素。最后，
迭代器的\texttt{collect}方法构建一个持有这些可变且不重叠切片的vector。

\medskip
现在我们可以让\texttt{crossbeam}库开始运作了：
\begin{minted}{rust}
    crossbeam::scope(|spawner| {
        ...
    }).unwrap();
\end{minted}

\medskip
参数\texttt{|spawner \{ ... \}|}为一个Rust闭包，该闭包的参数为\texttt{spawner}。需要注意的是，不同于函数用
\texttt{fn}关键字进行声明，我们不需声明闭包参数的类型，Rust会自动推导闭包的参数及返回类型。在这个例子中，
\texttt{crossbeam::scope}调用该闭包，并传递\texttt{spawner}作为该闭包可用作创建新线程的值。
\texttt{crossbeam::scope}函数等待所有这样的线程结束运行后再返回自身。此行为可让Rust确保这样的线程在脱离作用域后不会
再访问它们负责的\texttt{pixels}区域，并允许我们确认，若\texttt{crossbeam::scope}返回，则该图像的计算完成。若一切
顺利，\texttt{crossbeam::scope}返回\texttt{Ok(())}，但若生成的任意线程发生panic，它会返回\texttt{Err}。
此时，我们若对该\texttt{Result}调用\texttt{unwrap}，也会造成panic，而用户会收到一个报告。
\begin{minted}{rust}
    for (i, band) in bands.into_iter().enumerate() {
\end{minted}

\smallskip
这里我们对像素缓冲区的条带进行迭代。迭代器\texttt{into\_iter()}让循环体的每一次迭代均独享一个条带的所有权，以确保每次
仅有一个线程可对其进行写入，我们会在\redtxt{第五章}详细解释这是如何运作的。紧接着，适配器\texttt{enumerate}生成
元组，来将各vector元素和其对应的索引进行配对。
\begin{minted}{rust}
    let top = rows_per_band * i;
    let height = band.len() / bounds.0;
    let band_bounds = (bounds.0, height);
    let band_upper_left = 
        pixel_to_point(bounds, (0, top), upper_left, lower_right);
    let band_lower_right = 
        pixel_to_point(bounds, (bounds.0, top + height),
                        upper_left, lower_right);
\end{minted}

\smallskip
给定一个条带的索引和其实际尺寸后（回想一下，最后一个条带也许比其它的要短），我们便可以生成一个\texttt{render}所要求的
包围盒（bounding box），不过该包围盒仅包围该条带，并不包围整个图像。类似地，我们再利用\texttt{pixel\_to\_point}
函数找出该条带对应复平面的左上角和右下角。
\begin{minted}{rust}
    spawner.spawn(move |_| {
        render(band, band_bounds, band_upper_left, band_lower_right);
    });
\end{minted}

\smallskip
最后，我们创建一个线程来运行闭包\texttt{move |\_| \{ ... \}}。关键词\texttt{move}表示该闭包拥有它所使用变量的所有
权；特别是，只有该闭包能使用可变切片\texttt{band}。参数列表\texttt{|\_|}表示该闭包接收一个参数，但并不使用（即另一个
创建嵌套线程的spawner）

\medskip
就像我们之前提到那样，调用\texttt{crossbeam::scope}可确保所有线程均于它返回之前完成任务，这意味着将图像存入文件是安全
的。

\subsection{安全是无形的}
最后，我们最终得到的并行程序与我们用别的语言所写得的也许没有实质性区别：我们将像素缓冲区的片段分配给多处理器，让它们各自单独在自己的片段上工作，并且
当它们都完成时展示成果。那么Rust的并发支持究竟有什么特别之处呢？

\medskip
我们并没有在这里展示出我们{\bfseries{不应该}}写出的Rust程序。我们在本章中看到的代码正确地将缓冲区划分到了线程之间，然而在这些代码上有很多容易出
错的小细节，一旦改变可能使程序无法正确工作（从而引入数据竞争）；这些改变一个都通过不了Rust编译器的静态检查。C或C++编译器则会愉快地助你探索具有
微妙数据竞争程序的广辽空间；而Rust会在有问题时预先告诉你。

\medskip
在第\redtxt{四}和第\redtxt{五}章中，我们会描述Rust的内存安全规则。\redtxt{第十九章}会解释这些规则是如何确保并发的“无污染性”的。

\mksec{文件系统和命令行工具}
Rust已在命令行工具的世界中创出一席之地。作为一个现代，安全并且快速的系统编程语言，它提供一个工具箱，使程序员能用其构建流畅
的命令行接口，以对现有的工具进行功能复制或扩展。譬如，命令\texttt{bat}相当于一个支持语法高亮的\texttt{cat}且内置对分页
工具的支持，\texttt{hyperfine}可自动对任何能通过命令或管线运行的东西进行基准测试。

\medskip
虽然这些复杂的东西超出了本书范畴，但Rust可助你轻松涉足符合人体工程学命令行应用程序的世界。本节中，我们将向你展示如何构建
你自己的查找-替换工具，并配备多彩的输出与友好的错误消息。

\medskip
开始工作，我们要创建一个新Rust项目：
\begin{minted}{bash}
    $ cargo new quickreplace
         Created binary (application) `quickreplace` package
    $ cd quickreplace
\end{minted}

\smallskip
对于我们的程序，其需要两个外部crates：用以在终端里生成彩色输出的\texttt{text-colorizer}，以及具备实际查找-替换功能
的\texttt{regex}。和之前一样，我们将这些crates放进\mytextit{Cargo.toml}以告诉\texttt{cargo}我们需要它们：

前版本的可兼容扩展。所以如果我们用\texttt{1.2}版本的某crate来测试程序，那么该程序应仍能在该crate的\texttt{1.3}，

一个crate的\texttt{"1"}版本进行请求时，Cargo将使用该crate在\texttt{2.0}之前的最新可用版本。

\subsection{命令行接口}
该程序的接口十分简单。它接收四个参数：一个用以查找的字符串（或正则表达式），一个用以替换其的字符串（或正则表达式），输入
文件的名称，以及输出文件的名称。我们将从\mytextit{main.rs}开始，它附带一个包含这些参数的结构体：
\begin{minted}{rust}
    #[derive(Debug)]
    struct Arguments {
        target: String,
        replacement: String,
        filename: String,
        output: String,
    }
\end{minted}

\smallskip
属性\texttt{\#[derive(Debug)]}告诉编译器生成一些额外的代码，以方便我们在\texttt{println!}中用\texttt{\{:?\}}格
式化\texttt{Arguments}结构体。

\medskip
一旦用户输入错误的参数数字，按照惯例应打印一条简洁的消息来解释如何使用该程序。我们将用一个叫\texttt{print\_usage}的简
单函数来实现，然后将\texttt{text-colorizer}中的所有东西引进来，方便我们增添一些颜色：
\begin{minted}{rust}
    use text_colorizer::*;

    fn print_usage() {
        eprintln!("{} - change occurrences of one string into another",
                "quickreplace".green())
        eprintln!("Usage: quickreplace <target> <replacement> <INPUT> <OUTPUT>");
    }
\end{minted}

\smallskip
简单地在字符串尾部加上\texttt{.green()}将以字面意义产生一个由合适的ANSI转义码包覆的新字符串，以在终端模拟器中显示为
绿色。然后新字符串在被打印之前会被插入到剩余消息当中。

\medskip
现在我们可以对该程序的参数进行收集与处理：
\begin{minted}{rust}
    use std::env;

    fn parse_args() -> Arguments {
        
        let args: Vec<String> = env::args().skip(1).collect();

        if args.len() != 4 {
            print_usage();
            eprintln!("{} wrong number of arguments: expected 4, got {}.",
                "Error:".red().bold(), args.len());
            std::process::exit(1);
        }

        Arguments { 
            target: args[0].clone(), 
            replacement: args[1].clone(),
            filename: args[2].clone(), 
            output: args[3].clone() 
        }
    }
\end{minted}

\smallskip
为取得用户输入的参数，我们使用与先前例子相同的\texttt{args}迭代器。\texttt{.skip(1)}跳过该迭代器的首值（即被运行的
该程序的名称），因此结果仅包含命令行参数。

\medskip
方法\texttt{collect()}生成一个存储参数的\texttt{Vec}。接着我们检查参数个数是否正确，若不是，则打印一条消息并以一条
错误消息退出。我们再次对消息的一部分进行上色，并使用\texttt{.bold()}来使文本更粗。若参数个数正确，便将它们置入一个
\texttt{Arguments}结构体，并将其返回。

\medskip
接着我们将加上\texttt{main}函数，且仅调用\texttt{parse\_args}然后打印输出：
\begin{minted}{rust}
    fn main() {
        let args = parse_args();
        println!("{:?}", args);
    }
\end{minted}

\smallskip

\subsection{文件读写}
接下来，我们需要用一些方法来实际从文件系统获得数据以便我们进行处理，并在处理完成时将数据写回去。Rust有一套处理输入输出的
强劲工具，标准库设计者知道文件读写操作十分常见，所以他们有意将其设计得简易。我们需要做的仅是引入\texttt{std::fs}模块，
之后我们便能使用\texttt{read\_to\_string}和\texttt{write}函数：
\begin{minted}{rust}
    use std::fs;
\end{minted}

\smallskip
\texttt{std::fs::read\_to\_string}返回一个\texttt{Result<String, std::io::Error>}。如果该函数执行成功，它
将生成一个\texttt{String}。如果它执行失败，它会生成一个\texttt{std::io::Error}，其为用来表示I/O问题的标准库类型。
类似地，\texttt{std::fs::write}返回一个\texttt{Result<(), std::io::Error>}：若成功则为一个\texttt{()}单元
类型，若失败则是一个相同的错误。
\begin{minted}{rust}
    fn main() {
        let args = parse_args();
        
        let data = match fs::read_to_string(&args.filename) {
            Ok(v) => v,
            Err(e) => {
                eprintln!("{} failed to read from file '{}': {:?}",
                        "Error:".red().bold(), args.filename, e);
                std::process::exit(1);
            }
        };

        match fs::write(&args.output, &data) {
            Ok(_) => {},
            Err(e) => {
                eprintln!("{} failed to write to file '{}': {:?}",
                        "Error:".red().bold(), args.filename, e);
                std::process::exit(1);
            }
        }
    }
\end{minted}

\smallskip
这里我们使用事先写好的\texttt{parse\_args}函数，并将获得的文件名传递给\texttt{read\_to\_string}和\texttt{write}
。这些函数输出上的\texttt{match}对错误进行优雅的处理，并附带一点显眼的颜色来引起用户的注意。

\medskip
有了此更新的\texttt{main}函数，我们可以运行该程序并观察，当然，新旧文件的内容是完全相同的：
\begin{minted}{bash}
    $ cargo run "find" "replace" Cargo.toml Copy.toml
       Compiling quickreplace v0.1.0 (/home/jimb/rust/quickreplace)
        Finished dev [unoptimized + debuginfo] target(s) in 0.01s
         Running `target/debug/quickreplace find replace Cargo.toml Copy.toml`
\end{minted}

\smallskip
该程序{\bfseries{确实}}读入了输入文件\mytextit{Cargo.toml}，也确实写入了输出文件\mytextit{Copy.toml}，但由于
我们还未写任何实际执行查找和替换的代码，那么输出文件在内容上不会有任何变动。我们可以用\texttt{diff}命令简单地进行检查，
其并不会检测到任何区别：
\begin{minted}{bash}
    $ diff Cargo.toml Copy.toml
\end{minted}

\subsection{查找并替换}
本程序的最后一步是实现它的实际功能：查找并进行替换。为此，我们将使用\texttt{regex} crate，它可以编译并执行正则表达式。
该crate提供名叫\texttt{Regex}的结构体，代表一个经过编译的正则表达式。\texttt{Regex}有一个\texttt{replace\_all}
方法，正如它的名字：它在字符串中搜索所有匹配正则表达式的项并将它们各自替换为给定的替换字符串。我们可以将这个逻辑置入一个函数
中：
\begin{minted}{rust}
    use regex::Regex;
    fn replace(target: &str, replacement: &str, text: &str)
        -> Result<String, regex::Error> {
        let regex = Regex::new(target)?;
        Ok(regex.replace_all(text, replacement).to_string())
    }
\end{minted}

\smallskip
注意该函数的返回类型。就像我们之前使用的标准库函数，\texttt{replace}返回一个\texttt{Result}，只是这次的错误类型由
\texttt{regex} crate提供。

\medskip
\texttt{Regex::new}编译用户提供的正则式，若给定一个非法字符串它会执行失败。正如\texttt{Mandelbrot}程序那样，一旦
\texttt{Regex::new}失败我们便使用\texttt{?}进行短路，不过在此例中该函数返回一个针对\texttt{regex} crate的错误类
型。当正则式编译完成，其\texttt{replace\_all}方法将替换\texttt{text}中任何与给定的替换字符串匹配的项。

\medskip
若\texttt{replace\_all}查找成功，其返回一个匹配项均被替换为给定文本的新\texttt{String}。否则，
\texttt{replace\_all}返回一个指向源文本的指针，以避免不必要的内存分配和复制。然而在此例中，我们总是希望得到一份独立拷贝
，所以无论匹配成功与否，我们用\texttt{to\_string}方法取得包含源文本的\texttt{String}，且该\texttt{String}由
\texttt{Result::Ok}所包覆。

\medskip
现在，是时候将新函数纳入我们的\texttt{main}代码中了：
\begin{minted}{rust}
    fn main() {
        let args = parse_args();
        
        let data = match fs::read_to_string(&args.filename) {
            Ok(v) => v,
            Err(e) => {
                eprintln!("{} failed to read from file '{}': {:?}",
                          "Error:".red().bold(), args.filename, e);
                std::process::exit(1);
            }
        };
    
        let replaced_data = match replace(&args.target, &args.replacement, &data) 
        {
            Ok(v) => v,
            Err(e) => {
                eprintln!("{} failed to replace text: {:?}",
                          "Error:".red().bold(), e);
                std::process::exit(1);
            }    
        };
    
        match fs::write(&args.output, &replaced_data) {
            Ok(_) => {},
            Err(e) => {
                eprintln!("{} failed to write to file '{}': {:?}",
                          "Error:".red().bold(), args.filename, e);
                std::process::exit(1);
            }
        }
    }
\end{minted}
