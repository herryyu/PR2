\chapter{系统程序员也可拥有很棒的工具}

    \begin{flushright}

    \end{flushright}
\end{quote}
\begin{quote}
    “现在所有电脑都是并行化的...并行编程\textbf{才是}真正的编程。”
    \begin{flushright}

    \end{flushright}
\end{quote}
\begin{quote}
    “TrueType解析器的缺陷被一些国家的攻击者作为监视用途；所有软件都是安全敏感的。”
    \begin{flushright}
        ------\ttred{Andy Wingo}
    \end{flushright}
\end{quote}
我们将上述的三则开场白作为引子，从而翻开本书。不过让我们先从一个谜题讲起。下面的C程序的功能是什么呢？

\smallskip
\begin{minted}{c}
    int main(int argc, char **argv) {
        unsigned long a[1];
        a[3] = 0x7ffff7b36cebUL;
        return 0;
    }
\end{minted}
\medskip
今天早上，该程序在Jim的笔记本电脑上的输出为：

\smallskip
\begin{minted}{bash}
    undef: Error: .netrc file is readable by others.
    under: Remove password or make file unreadable by others.
\end{minted}

\smallskip
紧接着它就崩溃了。若你试着让它在你电脑上运行，它也许还会有别的表现。到底发生了什么呢？

\medskip
该程序是有缺陷的。数组\texttt{a}仅有一个元素的长度，则根据C语言标准，\texttt{a[3]}这样的操作是一个未定义行为
（undefined behavior）：

\smallskip
\begin{quote}
    “对于使用非可移植的或者错误的程序结构体，或使用错误数据这样的行为，本国际标准无任何明确要求。”
\end{quote}

\smallskip
未定义行为不仅仅会导致一个无法预测的结果：事实上，C标准明确允许这样的程序执行任何操作。在该程序中，我们将一个特定
16进制值存入该特定数组的第4个元素位，这破坏了函数调用栈，进而导致控制流未如预期那样使程序优雅地退出，而是从\texttt{main}函数返回，并跳入C标准库代码中间以执行“检索用户的主目录下一个文件中的密码”这个操作。目前不太顺利。

\medskip
C和C++包含大量规则来避免未定义行为。它们均具有一个共识：不要操作不该访问的内存、不要让算术操作溢出、不要除零等等。
然而编译器并不强制实施这些规则，它甚至不负责检查明显的违规操作。实际上，前述程序没有任何编译警告或者错误。那么避免
未定义行为的义务完全落在了你，也即程序员的肩上。

\medskip
以经验之谈，我们程序员在这个方面没有很好的记录。犹他大学的一名研究生Li Peng对C及C++编译器做了修改，由之编译出的程
序能报告它们是否执行了某些形式的未定义行为。结果他发现几乎所有程序都会，这包括那些对编码有着高标准的备受尊敬项目。假
设你能在C及C++中避免未定义行为，这便如你赢得了一盘棋，因为你确实知道具体规则。

\medskip
偶尔出现的奇怪信息或者崩溃也许是质量问题，但自从Morris蠕虫病毒于1988年使用了先前出现过的一种技术的变体且通过互联网
在计算机之间传播以来，不经意间的未定义行为也成为了安全漏洞的一个主要因素。

\medskip
那么C及C++将程序员置于了一个尴尬的位置：这些语言是系统编程的行业标准，但它们对于程序员的要求包含对一系列崩溃和安全问
题的保障。对于前述谜题的回答，引出了一个更大的问题：我们能否做得更好？

\mksec{让Rust为你扛起重担}
我们的回答由三则引用构成。第三则引述的报道称，人们于2010年发现一种叫Stuxnet的电脑蠕虫入侵了工业控制设备，在众多技
术手段中，其利用用于解析内置于文字处理文档中的TrueType字体的代码的未定义行为来控制受害者的电脑。毫无疑问，该代码作
者并不希望它以此方式被利用，这说明需要担心安全性的不只是操作系统和服务器：任何可能处理来自不受信任源数据的软件都可能
成为受攻击目标。

\medskip
Rust语言为你做出了简单承诺：若你的程序通过了编译器的检查，则它是无未定义行为的。悬空指针、双重free操作以及空指针解
引用均会在编译期被捕获。数组引用安全性由编译期和运行时的混合检查来保障，所以不会有缓冲区溢出：与Rust等价的不幸的C程
序则安全退出，并提示一条错误信息。

\medskip
除此之外，Rust旨在兼顾安全性又易于使用。为了对程序的行为做出更有力的保证，Rust相比C和C++对你的代码强制实行了更多限
制。但是该门语言总体而言灵活又富有表达能力，由Rust所编写代码的广度以及该语言被应用领域的范围证明了这一点。

\medskip
以经验之谈，若能信任一门语言能捕获更多错误，将鼓励我们尝试更多有野心的项目。当知道内存管理与指针合法性问题已被照管好
时，你修改复杂的大型程序的风险会更低。另外当一个bug的潜在后果中不包含破坏程序中的非关联部分时，调试会简单很多。

\medskip
当然了，仍有大量Rust无法检测的漏洞。但实际上，避免未定义行为将可观地改善开发角色，使之变得更好。

\mksec{被驯服的并行编程}
众所周知，并发在C及C++里是难以被正确使用的。开发者往往仅在单线程代码无法实现他们需要的性能时，才转向并发。但是开场第
二则引用表明，并行性对于现代机器极其重要，不能被当作一种最后手段。

\medskip
事实证明，Rust中保证内存安全的限制同时能保证Rust程序摆脱了数据竞争。只要数据不发生改变，你可自由在线程间共享它。确实
会发生变化的数据仅在启用了同步原语（synchronization primitives）后才可被访问。所有传统并发工具均可被找到：互斥器、
条件变量、通道以及原子操作等等。Rust单纯检查你是否正确使用它们。

\medskip
这使得Rust成为一门利用现代多核机器能力的优秀语言。Rust生态系统提供了超越普通并发原语（concurrency primitives）的
库，以及通过类似于“读-复制-更新”（Read-Copy-Update）的无锁同步机制帮助你于处理器池之间均匀分配复杂的负载，等等。

\mksec{然而Rust依旧很快}
终于轮到第一则引用了。Rust共享了Bjarne Stroustrup在论文“抽象和C++机器模型”中为C++所阐述的野心：

\smallskip
\begin{quote}
    “通常来说，C++的实现遵循零开销（zero-overhead）原则：你无需为你不使用的语言特性付出开销；更进一步：对于你使用
    的特性，你无法写出比它更好的代码。”
\end{quote}

\smallskip
系统编程通常致力于将机器推向极限。对于视频游戏，整个机器应致力于为玩家创造最好的体验。对于网络浏览器，浏览器的效能决定
了内容创作者可做事情的上限。在机器的固有限制范围内，必须将尽可能多的内存和处理器注意力置于内容本身。相同的原则也应用于
操作系统：内核应使机器的资源向用户开放，而不是供它自己使用。

\medskip
但当我们说Rust“很快”时，其真正含义是什么呢？人们用任意通用语言均可写出缓慢的代码，那么更精确的说法是，若你准备好投入精
力设计你的程序，以最大化利用底层机器的能力时，Rust可助你达到目标。该门语言以默认高效率的理念被设计出来，且赋予你能力以
管理内存如何被使用以及处理器的注意力如何被消耗。

\mksec{有了Rust，合作从未如此简单}
我们在本章的标题中隐藏了第四句话：“系统程序员也可拥有很棒的工具”，这是指Rust对代码共享和复用的支持。

\medskip
Rust的包管理和构建工具Cargo，使得调用别人在Rust公共包仓库，即crates.io上发布的库这件事变得异常简单。你只需将对应库
的名称以及要求的版本添加到一个文件中，Cargo会接管并将该库本身以及它依赖的所有库逐一下载，并将它们整体关联起来。你大可
将Cargo看作Rust对于NPM或者RubyGems而给出的答案，其重点是完善的版本管理和可复现构建（reproducible builds）
\footnote{一个构建（build）是可复现（reproducible）的，当且仅当给定相同的源代码、构建环境以及构建指令。此时任何人
都能逐位（bit-by-bit）地重新创建所有指定工作的相同副本。详情访问https://reproducible-builds.org/docs/definition/。}。
许多流行的Rust库提供从现成的序列化到HTTP客户、服务端以及现代图形接口的一切。

\medskip
进一步讲，该门语言本身也是被设计来支持合作的：Rust的特型（traits）和泛型（generics）允许你创建具有灵活接口的库，以便
它们能服务于不同的语境。同时Rust的标准库提供了一套核心的基础类型，它们为常见情形确立了共享公约，使不同的库更易于被共同
使用。
